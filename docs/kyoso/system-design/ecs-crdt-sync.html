1 · The mental model — two parallel stores

A peer holds two state stores for every replicated entity, kept in sync by a per-frame pipeline. The CRDT is the source of truth; the ECS is a projection onto it.

2 · What #[derive(SchemaSync)] generates

Given a Bevy component with field-level #[crdt(...)] annotations, the macro emits a sibling schema struct whose fields are the same but whose types are swapped for CRDT primitives, plus a SchemaSync impl that bridges the two.

Bevy component — what you write

crates/kyoso_figma/src/text.rs · 16
#[derive(Component, Default, Clone, Debug,
         PartialEq, Reflect, SchemaSync)]
#[reflect(Component, Default)]
#[schema(name = "Text")]
pub struct Text {
    #[crdt(sequence)]
    pub content: String,

    #[crdt(nested)]
    pub style: TypeStyle,

    pub fills: Vec<Paint>,   // default = LWW
}

Generated schema — what the macro emits

expanded — conceptual
#[derive(Clone, Debug, Default, PartialEq,
         kyoso_crdt::DeriveCrdt)]
pub struct TextSchema {
    pub content: Sequence<char>,            // sequence
    pub style:   TypeStyleSchema,           // nested
    pub fills:   LwwRegister<Vec<Paint>>,   // lww (default)
}

impl SchemaSync for Text {
    type Schema = TextSchema;
    const SCHEMA_NAME: &'static str = "Text";

    fn changes_against(&self, current: &TextSchema)
        -> Vec<TextSchemaMut> { /* per-field diff */ }

    fn write_back(&mut self, schema: &TextSchema) {
        /* per-field projection */
    }
}

2.1 Two macros, stacked

SchemaSync only generates the bridge. The schema struct itself becomes a real CRDT via a second macro, DeriveCrdt (crates/kyoso_crdt_derive), which the outer derive stamps onto it. That inner macro is what generates:

  • TextSchema::Mutation and TextSchema::Delta enums — one variant per field
  • impl Crdt for TextSchema — the mutate() dispatch
  • impl Lattice for TextSchema — the merge function
  • impl SchemaApply for TextSchema — apply_wire + install_state
  • impl IntoWireOp for TextSchema::Delta — (Path, WireDelta) lowering

2.2 The two bridge functions

This is the heart of the seam. Both functions are generated, one arm per field, with the shape determined by the field's CRDT kind.

generated by crates/kyoso_sync_derive/src/lib.rs · 174–497 
// === LWW arm (the `fills` field) ===

// changes_against:
if *current.fills.get().unwrap_or(&default.fills) != self.fills {
    out.push(TextSchemaMut::Fills(LwwMut::Set(self.fills.clone())));
}

// write_back:
if let Some(value) = schema.fills.get() {
    self.fills = value.clone();
}

// === Nested arm (the `style` field) — delegates to the inner type ===

// changes_against:
for inner_mut in SchemaSync::changes_against(&self.style, &current.style) {
    out.push(TextSchemaMut::Style(inner_mut));
}

// write_back:
SchemaSync::write_back(&mut self.style, &schema.style);

// === Sequence arm (the `content` field) ===

// changes_against:
let doc_view: Vec<char> = current.content.iter().into_iter().copied().collect();
for inner_mut in kyoso_sync::sequence_diff(doc_view, self.content.chars()) {
    out.push(TextSchemaMut::Content(inner_mut));
}

// write_back:
self.content = schema.content.iter().into_iter().copied().collect::<String>();

Order matters Inbound (route_typed_inbound) runs before outbound detection (detect_typed_changes) in the same frame. 5 · Outbound — Bevy edit → wire op

When you mutate component.field = new_value in a Bevy system, here's the path to the wire.

1

Bevy fires Changed<C> on the next frame.

fn user_system(mut q: Query<&mut Text>) {
    for mut text in &mut q {
        text.content.push_str(" world"); // Changed<Text> will fire
    }
}
2

detect_typed_changes picks it up, fetches the doc's current schema, and asks the bridge for the diff.

crates/kyoso_graph_sync/src/schema_sync.rs · 323
let Some(id)      = T::id_for(&index, entity)  else { continue };
let Some(current) = doc.schema(id)             else { continue };

// Reconnect-clobber guard — see §8.
if component.is_added() && *current != empty_schema { continue; }

let mutations = component.changes_against(current);
if mutations.is_empty() { continue; }
3

Each mutation is run through a throwaway copy of the schema to compute the typed Delta, which is then lowered to wire format.

let mut throwaway = current.clone();
for mutation in mutations {
    let op_id     = engine.mint_id();
    let mut state = kyoso_crdt::CausalState::new();
    let mut ctx   = kyoso_crdt::CausalContext::new(op_id, None, &mut state);

    let typed_delta       = throwaway.mutate(mutation, &mut ctx);
    let (inner_path, wire) = typed_delta.into_wire_op();
    // …
}
4

The path is namespaced with SCHEMA_NAME. This is how a single shared OpKind::SetNodeProperty multiplexes ops for any number of schema types — the wire-side Path looks like ["Text", "content"] rather than just ["content"]. 

let mut path = Path::field(C::SCHEMA_NAME);   // ["Text"]
for seg in inner_path.0 { path.0.push(seg); } // ["Text", "content"]

engine.enqueue(Op::new(op_id, T::build_op(id, path, wire)));
//                                ^^^^^^^^^
//      NodeTarget → SetNodeProperty
//      EdgeTarget → SetRefEdgeProperty
5

outbound_system (Phase 4) drains the engine's pending queue, postcard-encodes, and ships each op via the WebSocket bridge.

Why a throwaway clone? The SchemaDoc is only mutated by server-confirmed ops via apply_property_op in route_typed_inbound. Local mutations pre-stamping would desync the lattice if the server rejected them. Using a clone for mutate() keeps the doc clean and lets the server be authoritative.

6 · Inbound — wire op → Bevy projection

The mirror image. A SetNodeProperty op arrives over the WebSocket; it ends up writing the corresponding Bevy component a couple of system runs later.

1

Phase 1 — graph_inbound_system reads WsInbound events, calls engine.apply_remote for every op, and writes a RemoteOpApplied(op) Bevy message for anything the schema chain needs to consume.

2

route_typed_inbound filters those events for ops whose path head matches this C's SCHEMA_NAME, strips the head, and applies the rest to the doc.

crates/kyoso_graph_sync/src/schema_sync.rs · 385
for ev in events.read() {
    let op = &ev.0;
    let Some((target, path, delta)) = T::match_inbound(&op.kind) else { continue };
    let Some(PathSegment::Field(head)) = path.0.first()          else { continue };
    if head != C::SCHEMA_NAME { continue; }   // namespace filter

    let inner_path = Path(path.0[1..].to_vec());
    let stripped   = Op {
        id:   op.id,
        seq:  op.seq,
        kind: T::build_op(target, inner_path, delta.clone()),
    };
    doc.apply_property_op(&stripped)?;       // updates SchemaDoc<C::Schema>
}
3

project_typed_to_bevy notices doc.is_changed() and writes every indexed entity's schema back into its Bevy component via the bridge.

if !doc.is_changed() { return; }
for (entity, id) in T::pairs(&index).iter() {
    let Some(schema) = doc.schema(*id) else { continue };
    match components.get_mut(*entity) {
        Ok(mut component) => component.write_back(schema),
        Err(_) => {
            // Entity doesn't have C yet (snapshot-spawned without it).
            // Queue a command that inserts C::default() then write_back.
            commands.queue(InsertSchemaProjected::<C> { entity: *entity, schema: schema.clone(), .. });
        }
    }
}

Inbound side-effects

  • SchemaDoc<C::Schema> mutates → is_changed() fires
  • Bevy component fields get overwritten by write_back
  • Changed<C> would fire next frame…
  • …but changes_against returns empty (doc already matches)
←

Outbound echo-guard

  • No new ops emitted from a remote write
  • Echo prevention is stateful comparison, not a "just-projected" set
  • Tried-and-removed: explicit echo bookkeeping (subsumed by this check)

7 · Worked example — Text

Tracing each of Text's three fields end-to-end. The component lives at crates/kyoso_figma/src/text.rs:16, with the nested type at typestyle.rs:12.

content: String sequence

Bevy edit

text.content = "Hi world".into();
// (previous was "Hello")

Wire ops produced

SetNodeProperty {
  target: id,
  path:   ["Text", "content"],
  delta:  SequenceDelta::Delete { positions… }
}
SetNodeProperty {
  target: id,
  path:   ["Text", "content"],
  delta:  SequenceDelta::Insert { chars… }
}

sequence_diff computes a prefix/suffix-aware character-level diff against the doc's Sequence<char>. The CRDT itself is RGA-style (Replicated Growable Array) so concurrent edits at the same position from different peers interleave deterministically rather than clobber.

style: TypeStyle nested

Bevy edit

text.style.font_size = 24.0;
// font_family, font_weight,
// line_height unchanged

Wire op produced

SetNodeProperty {
  target: id,
  path:   ["Text", "style", "font_size"],
  delta:  LwwDelta::Set(24.0)
}

Recursion: Text::changes_against delegates to TypeStyle::changes_against for the style field, wraps each returned mutation in TextSchemaMut::Style(inner), and lets the path segments accumulate naturally as the Delta is lowered. Two peers concurrently changing different sub-fields (one edits font_size, one edits font_family) merge without conflict — that's the whole point of #[crdt(nested)] vs. wrapping the entire struct in LWW.

fills: Vec<Paint> lww (default)

Bevy edit

text.fills = vec![Paint::Solid(red)];
// concurrent peer sets fills to
// vec![Paint::Solid(blue)]

Wire op produced

SetNodeProperty {
  target: id,
  path:   ["Text", "fills"],
  delta:  LwwDelta::Set(vec![Paint::Solid(red)])
}

Whole-list LWW: only one peer's Vec survives, resolved by GlobalSeq on the server. If you wanted per-element conflict-free merging instead, you'd use #[crdt(or_set)] (membership) or a sequence (if order matters).

8 · The reconnect-clobber guard

This is the bug that catches everyone who builds a system like this. It's worth understanding because it explains a non-obvious condition in detect_typed_changes.

The failure mode

A peer reconnects. The server sends a Welcome snapshot. Locally:

  1. hydrate_typed_schemas populates every SchemaDoc with the real per-id state.
  2. project_snapshot queues commands.spawn(N::default()) for each node — but only the structural marker is in the bundle.
  3. Bevy's Add observer fires synchronously; any #[require(...)] components (Transform, Mesh3d, …) get auto-inserted with Default::default().
  4. That trips Changed<C> before project_typed_to_bevy has a chance to write the hydrated values back.
  5. A naive changes_against sees "component is default, doc has real values" and emits ops setting every field to its local default — broadcasting them to the room and wiping out the canonical state on every peer.

The discriminator

crates/kyoso_graph_sync/src/schema_sync.rs · 362
// Snapshot-hydrated entities have `component.is_added()` AND the doc already
// holds non-default state (it was just installed by `hydrate_typed_schemas`).
// User-spawned entities also have `is_added()` but the doc is still at lattice
// bottom for that id, so they fall through to the regular detect path and the
// initial values do replicate.
if component.is_added() && *current != empty_schema {
    continue;
}
Caseis_added()doc stateBehaviour
Snapshot-hydrated entity true non-default Skip — let projection overwrite component
User-spawned entity true default (empty) Continue — emit ops for the initial field values
Existing entity, local edit false any Continue — emit ops for the diff
Existing entity, just got remote write false matches component Continue, but changes_against returns vec![]
Why PartialEq on Schema is required The bound S: PartialEq on SchemaSync::Schema isn't decoration — it's exactly so we can distinguish "the doc holds real hydrated state" from "the doc is still at its empty/default lattice bottom." Without it, this guard is unimplementable.

9 · Late-join hydration

The server doesn't know the concrete S for any peer's schema doc — it stores everything as OpaqueRecord, a BTreeMap<Path, OpaqueValue> of raw per-primitive bytes. On Welcome the joining peer has to demux those blobs back into the right SchemaDoc<S> resources.

At app-build time each SchemaSyncedNodeComponentPlugin registers a monomorphized hydrate_schema_doc::<C> under the key (TargetKind, C::SCHEMA_NAME):

crates/kyoso_graph_sync/src/schema_sync.rs · 287
fn hydrate_schema_doc<C>(world: &mut World, target: CrdtId, path: Path, field: OpaqueValue)
where
    C: SchemaSync,
{
    let mut doc = world.resource_mut::<SchemaDoc<C::Schema>>();
    doc.ensure_schema(target);
    if let Some(state) = doc.schema_mut(target) {
        // SchemaApply::install_state — the other half of derive(Crdt).
        // Decodes the opaque bytes via the schema's known type.
        state.install_state(&path, field)?;
    }
}

At Welcome time the inbound system walks every (target, opaque_value) pair from the snapshot, looks up the hydrator by (TargetKind, schema_name), and dispatches. The hydrator's monomorphization knows the concrete S, so install_state can decode the bytes via the typed SchemaApply impl.

Summary — the seam in one screen

What the macro does

  1. Reads #[crdt(...)] on each field.
  2. Emits a sibling CSchema struct with each field's type swapped for the matching CRDT primitive.
  3. Stamps kyoso_crdt::DeriveCrdt on it — that's what makes the schema a real CRDT.
  4. Emits impl SchemaSync for C with the two bridge functions.

What the runtime does

  1. Stores per-id schema state in SchemaDoc<C::Schema> resources.
  2. Each frame: hydrate slots, route inbound ops into the doc, project doc to component, detect Bevy edits and emit ops.
  3. Wire format namespaces with SCHEMA_NAME so all schemas share one OpKind.
  4. Server stores everything as opaque bytes; clients hydrate via a registered HydratorFn per (TargetKind, SCHEMA_NAME).
